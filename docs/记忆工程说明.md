# 记忆工程说明（2026-07-10 一轮改造）

> 这份文档是这一整套记忆分流工程的总图：做了什么、代码在哪、怎么生效、哪些还没验证。
> 设计讨论的来龙去脉在当天对话里，这里只记结论。

---

## 一、设计总图

参照 ebbingflow 的分层思路（Event 事件层 → Episode 叙事层 → Saga 主线层），对齐我们自己的库：

| 层 | 我们的对应物 | 特性 |
|----|------------|------|
| 事件层 | 动态库（dynamic） | 原子事件，有遗忘曲线，可检索 |
| 叙事/独白层 | 日记（MD 文件） | 有质感的细节、感受、独白，不追求全 |
| 主线层 | chronicle（周/月画像） | DS 生成，只读 |
| 永久层 | 核心库（core）+ 房间 | 不衰减；「进度页」也住这里 |

这轮解决的核心问题：**日记太杂（在替动态库干活）、日记和动态库不会同时写、约定类内容检索全靠噪音**。

## 二、六个模块

### 1. 日记约定 + 节点切分（手动按钮）

- 约定：[diary_convention.md](diary_convention.md) —— 四步法（梗概/细节/感受/独白），细节感受挂【节点】，切分规则和示例都在里面
- 切分：管理面板日记列表每篇旁的**【切分】按钮**，按下才调 DeepSeek，按【】节点机械切分（原文拼接不改写），每条前缀 `[日期 时间]【节点名】`，落动态库；【整体】感受和独白不入库
- 幂等：重按 = 重切，靠 metadata 的 `diary_file` 字段先删旧条目
- 代码：`claude_memory.py` 的 `claude_split_diary_to_dynamic()`；端点 `main.py` 的 `POST /admin/diary-split`；按钮 `index.html` 的 `erikSplitDiary()`

### 2. 固定注入词条表

- 配置：[pinned_memories.json](pinned_memories.json) —— 多词条，每词条自己的触发词组 + 内容来源（`ids` 走 Chroma 精确取 / `file` 读仓库文件）
- 行为：📎 开启 + 消息命中触发词 → 该词条整块注入，跳过模糊检索；**每 session 每词条只注入一次**；多词条可同时命中各注各的
- 当前词条：床边描写备忘、日记约定、亲密地图、AK炮机说明书、月月兔兔说明书、葬礼与遗产、母亲家族、共读批注约定
- 代码：`cc_ws_gateway.py` 的 `match_pinned_entries()` / `fetch_pinned_injection()`，session 上的 `pinned_injected` set

### 3. 标签六类白名单

- 集合定稿：**亲密 / 情感 / 日常 / 冲突 / 纪念日 / 思考**，与检索权重表同源
- 权重：纪念日 1.5 > 冲突/情感/亲密 1.3 > 思考 1.1 > 日常 1.0（`claude_memory.py` 的 `CATEGORY_WEIGHTS`）
- 写入归一：`store_dynamic` 传集合外标签按词根归一、默认日常（`claude_mcp.py` 的 store_dynamic 分支）；DS 压缩 prompt 三处选项已换六类（`claude_memory.py`，搜"类型：（"）
- 存量 83 条**不迁移**：打分是子串匹配，旧花式标签含六类词根的照常蹭权重，其余等关键词精准命中

### 4. 共读约定 + 进度页

- 约定：[coreading_convention.md](coreading_convention.md) —— 批注纪律 + 进度页规则
- 进度页 = 「可覆盖的一页纸」：**状态不是事件**，不往库里堆新条目，用 `edit_core` 反复更新同一条核心记忆
- 每本书一条（存「Switch/读书进度」房间），每次共读结束更新章节概括给下一个 session 接上文；新书才 `store_core` 新建；星露谷单页（「Switch」房间）
- 存储位置：**直接在线上 Chroma 核心库**（通过 palace MCP 存的，不依赖代码部署），面板核心 tab 可见可编辑

### 5. get_by_id 精准通道 + 后台 prompt 指引

- `palace(cmd="get_by_id", data={ids:[...]})`：按 id 整块取核心记忆，零噪音不打分——就是 📎 词条注入的同一条底层通道开放给 CC 自取（`claude_mcp.py`）
- pebbling/desire 触发时的取用指引挂在三处 prompt：`cc_ws_gateway.py` 的 `build_pebbling_prompt()` free 分支、`desire_gateway.py` 的 `build_desire_pebbling_prompt()` 和 `build_desire_proactive_prompt()`
- 指引逻辑：备忘 id 列表 cat `docs/pinned_memories.json` → `get_by_id`；进度页走 `list_room`；取过一次不重复取；日常回忆照常 `search`

### 6. list_room 动态房间

- 房间列表不再硬编码六间，动态来自 Chroma 的 `folder` 字段——`store_core` 传新 `folder`（支持子路径如 `Switch/读书进度`）即建新房间
- 输出加了 `id:` 字段，供 `edit_core` 更新进度页
- 代码：`claude_memory.py` 的 `claude_list_room()`

## 五、已知边界 / 遗留

- `Obsidian_Core` 目录**不在持久卷里**：`store_core` 落的 MD 文件随 redeploy 蒸发，Chroma 向量才是权威。想留 MD 副本可在 Coolify 加卷（destination `/app/Obsidian_Core`），但加之前先查 `sync_claude_memory.py` 的孤立删除对账逻辑会不会误删手动存的记忆（`restore_core.py` 的存在说明历史上出过事故）
- 容器内的 `docs/pinned_memories.json` 是构建时版本，palace 侧用不到它（词条表只有网关读，网关在 VPS 上 git pull 即最新）——不算问题，记一笔防混淆
- "get_by_id 只取一次"是 prompt 约定不是硬机制（palace 无 session 状态，硬拦需要引入状态机，不值）
- 贴表情没做词条：无触发词可言，硬做等于常驻注入
